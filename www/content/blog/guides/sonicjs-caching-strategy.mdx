# Understanding SonicJS Three-Tiered Caching Strategy

<TLDRBox>
**TL;DR** — SonicJS uses a three-tiered caching system: Memory (sub-1ms), KV (5-15ms), and D1 (20-50ms). Content is checked in order from fastest to slowest, with automatic population of faster tiers on cache misses.

**Key Stats:**
- Memory cache: Under 1ms access time
- KV cache: 5-15ms globally distributed
- 90-95% cache hit rate for well-configured sites
- Only 5-10% of requests hit the D1 database
</TLDRBox>

Performance is at the heart of SonicJS. To deliver consistent sub-15ms response times globally, SonicJS implements a sophisticated three-tiered caching strategy. This guide explains how it works and how to configure it for your needs.

## The Three Tiers

SonicJS caching operates across three layers:

```
Request → Memory Cache → KV Cache → D1 Database
              ↓              ↓           ↓
           <1ms           5-15ms      20-50ms
```

### Tier 1: Memory Cache

The fastest layer, storing frequently accessed content in V8 isolate memory.

**Characteristics:**
- **Speed**: Sub-millisecond access
- **Scope**: Per-isolate (not shared across data centers)
- **Lifetime**: Duration of the isolate (typically minutes)
- **Size**: Limited by isolate memory

### Tier 2: KV Cache

Cloudflare KV provides globally distributed, persistent caching.

**Characteristics:**
- **Speed**: 5-15ms globally
- **Scope**: Shared across all edge locations
- **Lifetime**: Configurable TTL
- **Size**: Up to 25MB per value

### Tier 3: D1 Database

The source of truth, providing SQLite at the edge.

**Characteristics:**
- **Speed**: 20-50ms
- **Scope**: Replicated globally
- **Lifetime**: Persistent
- **Size**: 2GB+ per database

## How Caching Works

### Read Path

When a request comes in:

```typescript
async function getContent(key: string) {
  // 1. Check memory cache first
  const memoryResult = memoryCache.get(key)
  if (memoryResult) {
    return memoryResult // <1ms
  }

  // 2. Check KV cache
  const kvResult = await env.CACHE.get(key, 'json')
  if (kvResult) {
    memoryCache.set(key, kvResult) // Populate memory
    return kvResult // 5-15ms
  }

  // 3. Query D1 database
  const dbResult = await db.query(...)
  if (dbResult) {
    await env.CACHE.put(key, JSON.stringify(dbResult), { ttl })
    memoryCache.set(key, dbResult)
    return dbResult // 20-50ms
  }

  return null
}
```

### Write Path

When content is updated:

```typescript
async function updateContent(key: string, data: any) {
  // 1. Update D1 database
  await db.update(...)

  // 2. Invalidate KV cache
  await env.CACHE.delete(key)

  // 3. Memory cache expires naturally
  // (or explicitly invalidated if supported)
}
```

## Configuration Options

### Basic Setup

SonicJS exposes a `CacheService` from `@sonicjs-cms/core`. Built-in pre-configured instances cover the most common namespaces:

```typescript
import { getCacheService, CACHE_CONFIGS, CacheService } from '@sonicjs-cms/core'

// Use a pre-configured instance (api: 5min TTL, content: 5min, collection: 10min)
const cache = getCacheService(CACHE_CONFIGS.api)

// Or create a custom instance
const myCache = new CacheService({ ttl: 3600, keyPrefix: 'my-plugin' })
```

### Content-Specific TTLs

Configure TTL per `CacheService` instance:

```typescript
import { CacheService } from '@sonicjs-cms/core'

// Short TTL for frequently updated content
const postsCache = new CacheService({ ttl: 60, keyPrefix: 'posts' })

// Long TTL for rarely changed reference data
const categoriesCache = new CacheService({ ttl: 86400, keyPrefix: 'categories' })
```

### Cache Headers

API responses include cache status headers set by the route handler:

```http
X-Cache-Status: HIT
X-Cache-Source: memory
X-Cache-TTL: 287
```

## Cache Invalidation Strategies

### Time-Based (TTL)

The simplest approach - content expires after a set time:

```typescript
{ ttl: 300 } // Cache for 5 minutes
```

### Pattern-Based

`cache.invalidate(pattern)` matches keys by glob-style pattern:

```typescript
// Invalidate all content-filtered keys
await cache.invalidate('content-filtered:*')

// Invalidate a specific key
await cache.invalidate('api:blog-posts:abc123')
```

### Event-Based

Invalidate on content write events using the `content:save` hook:

```typescript
import { definePlugin, getCacheService, CACHE_CONFIGS } from '@sonicjs-cms/core'

const cache = getCacheService(CACHE_CONFIGS.content)

export const myPlugin = definePlugin({
  onBoot(ctx) {
    ctx.hooks.on('content:save', async (content) => {
      await cache.invalidate(`content-filtered:*`)
    })
  },
})
```

## Performance Benchmarks

Real-world performance across cache tiers:

| Scenario | Memory Hit | KV Hit | D1 Query |
|----------|-----------|--------|----------|
| Simple content | 0.5ms | 8ms | 25ms |
| Complex query | 1ms | 12ms | 45ms |
| With relations | 1.5ms | 15ms | 60ms |

### Cache Hit Rates

Typical hit rates for well-configured caching:

- **Memory**: 60-80% for hot content
- **KV**: 90-95% for all cached content
- **D1**: Only ~5-10% of requests

## Best Practices

### 1. Set Appropriate TTLs

```typescript
// Static content: Long TTL
'/api/pages': { ttl: 86400 }, // 24 hours

// Dynamic listings: Short TTL
'/api/posts': { ttl: 60 }, // 1 minute

// Real-time data: No caching
'/api/analytics': { ttl: 0 },
```

### 2. Use getOrSet for Computed Values

`CacheService.getOrSet` reads from cache and computes on miss — no stale-while-revalidate but zero boilerplate:

```typescript
const result = await cache.getOrSet(
  cache.generateKey('posts', slug),
  async () => await fetchFromDb(slug),
  300 // TTL in seconds
)
```

### 3. Warm Critical Caches

Pre-populate caches for critical content:

```typescript
// Warm cache on deploy
async function warmCache() {
  const posts = await db.getAllPublishedPosts()
  for (const post of posts) {
    await cache.set(`post:${post.slug}`, post)
  }
}
```

### 4. Monitor Cache Performance

Use `getWithSource` to track cache effectiveness in custom routes:

```typescript
const { hit, data, source, ttl } = await cache.getWithSource<Post[]>(key)
console.log(`Cache ${hit ? 'HIT' : 'MISS'} from ${source}, TTL: ${ttl}s`)
```

API responses also emit `X-Cache-Status`, `X-Cache-Source`, and `X-Cache-TTL` headers automatically.

## Debugging Cache Issues

### Check Cache Status

API responses include cache metadata:

Cache status is in response headers:

```http
X-Cache-Status: HIT
X-Cache-Source: memory
X-Cache-TTL: 3555
```

### Force Cache Bypass

For debugging, bypass the cache:

```bash
curl -H "Cache-Control: no-cache" https://api.example.com/content/posts
```

### View Cache Keys

List cached keys:

```bash
wrangler kv:key list --namespace-id YOUR_KV_ID
```

## Advanced Patterns

### Content-Aware Caching

Use separate `CacheService` instances with different TTLs per content type:

```typescript
import { CacheService } from '@sonicjs-cms/core'

const newsCache = new CacheService({ ttl: 30, keyPrefix: 'news' })
const pageCache = new CacheService({ ttl: 86400, keyPrefix: 'pages' })

// In a plugin route:
const result = await newsCache.getOrSet(
  newsCache.generateKey('article', id),
  () => fetchArticle(id),
  30
)
```

## Key Takeaways

- Three-tiered caching provides sub-15ms response times
- Memory cache is fastest but per-isolate
- KV cache provides global persistence
- D1 is the source of truth, accessed only on cache miss
- Configure TTLs based on content update frequency
- Use tag-based invalidation for related content
- Monitor cache hit rates for optimization
